# Title

# Conclusion: Creating the Graphics Pipeline with Vulkan and Zig 0.16.0

## Overview

The previous lessons prepared the resources required to describe a draw:

1. Koba creates an SDL3 window.
2. Vulkan creates an instance.
3. SDL3 creates a Vulkan surface.
4. Koba selects a physical device.
5. Koba creates a logical device and graphics queue.
6. Koba creates a swap chain.
7. Koba retrieves swap-chain images.
8. Koba creates image views.
9. Koba loads shader code and creates a shader module.
10. Koba describes the vertex and fragment shader stages.

This conclusion connects those pieces by creating the Vulkan graphics pipeline.

A graphics pipeline is the object that combines shader stages with fixed-function rendering state:

```text
shader stages
    +
vertex input
    +
primitive assembly
    +
viewport and scissor
    +
rasterization
    +
multisampling
    +
color blending
    +
pipeline layout
    +
dynamic-rendering format
    |
    v
graphics pipeline
```

The C++ source uses Vulkan-Hpp's `vk::StructureChain` and RAII handles:

```cpp
vk::StructureChain<
    vk::GraphicsPipelineCreateInfo,
    vk::PipelineRenderingCreateInfo
>
```

Koba uses raw C Vulkan bindings generated by `addTranslateC`. Therefore, the translation must:

- create `vk.VkGraphicsPipelineCreateInfo` directly,
- create `vk.VkPipelineRenderingCreateInfo` directly,
- connect them with `.pNext`,
- call `vk.vkCreateGraphicsPipelines`,
- store the resulting `vk.VkPipeline`,
- destroy it explicitly during cleanup.

The pipeline is created inside the existing `HelloTriangleApplication`. No proxy objects, alternate application type, or new module is introduced.

## Concepts & Explanations

### Why the graphics pipeline is the final initialization step

A graphics pipeline depends on several objects and descriptions created earlier:

```text
logical device
    |
    +-- shader module
    +-- pipeline layout
    +-- swap-chain format
    +-- swap-chain extent
    +-- image views
```

The pipeline does not directly own the swap-chain images or image views. However, it must be compatible with the format of the images that will receive its color output.

The complete rendering sequence will eventually look like this:

```text
acquire swap-chain image
    ↓
begin dynamic rendering using that image
    ↓
bind graphics pipeline
    ↓
draw vertices
    ↓
end dynamic rendering
    ↓
present swap-chain image
```

Creating the pipeline now completes the resource-description phase. The next lessons can focus on command buffers, synchronization, image layouts, and presentation.

### What `vk::StructureChain` was doing in C++

The C++ source uses:

```cpp
vk::StructureChain<
    vk::GraphicsPipelineCreateInfo,
    vk::PipelineRenderingCreateInfo
>
```

This is a convenience wrapper around Vulkan's `pNext` mechanism.

The equivalent raw Vulkan relationship is:

```text
VkGraphicsPipelineCreateInfo
    .pNext ───────────────► VkPipelineRenderingCreateInfo
```

In Zig, build the structures separately:

```zig
var rendering_info = vk.VkPipelineRenderingCreateInfo{
    // ...
};

var pipeline_create_info = vk.VkGraphicsPipelineCreateInfo{
    .pNext = &rendering_info,
    // ...
};
```

The `rendering_info` value must remain alive while `vk.vkCreateGraphicsPipelines` reads it. Keeping both structures in the same function satisfies that requirement.

This is one of the important differences between a wrapper and a raw binding: the wrapper hides the pointer chain, while Zig makes it explicit.

### Dynamic rendering replaces a render pass here

The source uses:

```cpp
.renderPass = nullptr
```

and supplies a `vk::PipelineRenderingCreateInfo` structure through the chain.

That means this pipeline uses Vulkan dynamic rendering rather than a traditional `VkRenderPass` object. The color format is supplied with:

```zig
.colorAttachmentCount = 1,
.pColorAttachmentFormats = &color_format,
```

The format must match the swap-chain format selected earlier:

```zig
self.swap_chain_surface_format.format
```

This matters because the fragment shader produces color values that must be written into the swap-chain image. A mismatch between pipeline format and rendering attachment format can cause pipeline creation or rendering validation errors.

Dynamic rendering reduces the amount of render-pass boilerplate, but it does not remove the need to describe attachment formats correctly.

### The pipeline states are deliberately simple

This first pipeline uses a minimal triangle configuration:

- no vertex-buffer input,
- triangle-list primitive assembly,
- one viewport,
- one scissor rectangle,
- filled polygon rasterization,
- back-face culling,
- counter-clockwise front faces,
- one-sample multisampling,
- disabled blending,
- dynamic viewport and scissor state.

The shaders use `SV_VertexID`, so the vertex-input state is empty:

```zig
.vertexBindingDescriptionCount = 0,
.vertexAttributeDescriptionCount = 0,
```

This means the first triangle can be drawn without creating a vertex buffer. Later lessons can add real vertex data and update this state.

### Dynamic viewport and scissor state

The pipeline declares viewport and scissor state, but marks both values as dynamic:

```zig
.dynamicStateCount = 2,
.pDynamicStates = &dynamic_states,
```

The command buffer will later set them with commands such as:

```zig
vkCmdSetViewport
vkCmdSetScissor
```

The trade-off is:

- dynamic state makes resize handling and command recording more flexible,
- static state can make pipeline configuration more explicit and may be preferable when values rarely change.

For a windowed game engine, dynamic viewport and scissor state are useful because the drawable size can change when the window is resized or moved between displays.

### Why pipeline layout is required

The pipeline layout describes resources that shaders may access, such as:

- descriptor sets,
- uniform buffers,
- sampled images,
- storage buffers,
- push constants.

The current shaders do not use any external resources, but Vulkan still requires a pipeline layout. For this first pipeline, the layout can be empty.

The graphics pipeline and pipeline layout are separate Vulkan objects:

```text
pipeline layout
    describes shader resource interfaces

graphics pipeline
    describes how shaders and rasterization operate
```

The graphics pipeline must be destroyed before the logical device. The pipeline layout should also be destroyed before the device.

### Every Vulkan result must be checked

The raw binding returns a `VkResult` from:

```zig
vk.vkCreateGraphicsPipelines(...)
```

The result must be compared explicitly:

```zig
if (result != vk.VK_SUCCESS) {
    return error.FailedToCreateGraphicsPipeline;
}
```

Pipeline creation can fail because of:

- invalid SPIR-V,
- a wrong shader entry-point name,
- unsupported device features,
- incompatible dynamic-rendering formats,
- invalid fixed-function state,
- a missing or invalid pipeline layout.

Returning a named Zig error is more useful than continuing with a null pipeline handle.

## Code Translation Sections

### Add graphics-pipeline state

Add these fields to the existing `HelloTriangleApplication` struct if they are not already present:

```zig
pipeline_layout: vk.VkPipelineLayout = null,
graphics_pipeline: vk.VkPipeline = null,
```

They belong near the shader and swap-chain fields:

```zig
swap_chain: vk.VkSwapchainKHR = null,
swap_chain_images: []vk.VkImage = &.{},
swap_chain_surface_format: vk.VkSurfaceFormatKHR = undefined,
swap_chain_extent: vk.VkExtent2D = undefined,
swap_chain_image_views: []vk.VkImageView = &.{},

shader_module: vk.VkShaderModule = null,
pipeline_layout: vk.VkPipelineLayout = null,
graphics_pipeline: vk.VkPipeline = null,
```

The nullable handle has a clear meaning:

- `null`: the object has not been created or has already been destroyed,
- non-null: Koba owns the Vulkan object and must destroy it.

### Extend `initVulkan`

The graphics pipeline must be created after the shader module, pipeline layout, swap-chain format, and image views are available.

Extend the existing entrypoint:

```zig
fn initVulkan(self: *HelloTriangleApplication) !void {
    try self.createInstance();
    try self.setupDebugMessenger();
    try self.createSurface();
    try self.pickPhysicalDevice();
    try self.createLogicalDevice();
    try self.createSwapChain();
    try self.createImageViews();
    try self.createShaderModules();
    try self.createPipelineLayout();
    try self.createGraphicsPipeline();
}
```

If the existing project already creates the pipeline layout elsewhere, keep that existing call and do not create it twice.

The important dependency is:

```text
shader module + pipeline layout + swap-chain format
    |
    v
graphics pipeline
```

### Create the pipeline layout if it does not exist

The current shaders do not use descriptor sets or push constants, so an empty pipeline layout is sufficient:

```zig
fn createPipelineLayout(self: *HelloTriangleApplication) !void {
    if (self.device == null) {
        return error.DeviceNotCreated;
    }

    if (self.pipeline_layout != null) {
        return error.PipelineLayoutAlreadyCreated;
    }

    const create_info = vk.VkPipelineLayoutCreateInfo{
        .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_LAYOUT_CREATE_INFO,
        .pNext = null,
        .flags = 0,
        .setLayoutCount = 0,
        .pSetLayouts = null,
        .pushConstantRangeCount = 0,
        .pPushConstantRanges = null,
    };

    const result = vk.vkCreatePipelineLayout(
        self.device,
        &create_info,
        null,
        &self.pipeline_layout,
    );

    if (result != vk.VK_SUCCESS) {
        std.log.err("Failed to create Vulkan pipeline layout", .{});
        return error.FailedToCreatePipelineLayout;
    }

    std.log.debug("Created Vulkan pipeline layout", .{});
}
```

The raw C binding uses:

```zig
vk.vkCreatePipelineLayout(...)
```

Do not replace it with a wrapper-style call such as:

```zig
self.device.createPipelineLayout(...)
```

### Describe the shader stages

The shader module created in the previous lesson contains both entry points:

- `vertMain`,
- `fragMain`.

Create the two stage descriptions locally in `createGraphicsPipeline`:

```zig
const shader_stages = [_]vk.VkPipelineShaderStageCreateInfo{
    .{
        .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_SHADER_STAGE_CREATE_INFO,
        .pNext = null,
        .flags = 0,
        .stage = vk.VK_SHADER_STAGE_VERTEX_BIT,
        .module = self.shader_module,
        .pName = "vertMain",
        .pSpecializationInfo = null,
    },
    .{
        .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_SHADER_STAGE_CREATE_INFO,
        .pNext = null,
        .flags = 0,
        .stage = vk.VK_SHADER_STAGE_FRAGMENT_BIT,
        .module = self.shader_module,
        .pName = "fragMain",
        .pSpecializationInfo = null,
    },
};
```

The entry-point names must match the functions compiled into the SPIR-V module. The filename is not an entry point.

Before constructing these stages, validate the handles:

```zig
if (self.shader_module == null) {
    return error.ShaderModuleNotCreated;
}

if (self.pipeline_layout == null) {
    return error.PipelineLayoutNotCreated;
}
```

### Configure vertex input

The current vertex shader obtains positions and colors from `SV_VertexID`. No vertex buffer is needed yet:

```zig
const vertex_input = vk.VkPipelineVertexInputStateCreateInfo{
    .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_VERTEX_INPUT_STATE_CREATE_INFO,
    .pNext = null,
    .flags = 0,
    .vertexBindingDescriptionCount = 0,
    .pVertexBindingDescriptions = null,
    .vertexAttributeDescriptionCount = 0,
    .pVertexAttributeDescriptions = null,
};
```

This is a deliberate temporary simplification. When Koba begins loading mesh data, this state will describe:

- the stride between vertices,
- the input rate,
- the location of each vertex attribute,
- the attribute format,
- the byte offset of each attribute.

### Configure primitive assembly

The first renderer draws independent triangles:

```zig
const input_assembly = vk.VkPipelineInputAssemblyStateCreateInfo{
    .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_INPUT_ASSEMBLY_STATE_CREATE_INFO,
    .pNext = null,
    .flags = 0,
    .topology = vk.VK_PRIMITIVE_TOPOLOGY_TRIANGLE_LIST,
    .primitiveRestartEnable = vk.VK_FALSE,
};
```

`VK_PRIMITIVE_TOPOLOGY_TRIANGLE_LIST` interprets every group of three vertices as one triangle.

Other topologies, such as triangle strips, can reuse vertices between adjacent triangles. Triangle lists are easier to understand initially, but may use more vertex data.

### Configure the viewport and scissor

The viewport maps normalized device coordinates to framebuffer pixels:

```zig
const viewport = vk.VkViewport{
    .x = 0.0,
    .y = 0.0,
    .width = @floatFromInt(self.swap_chain_extent.width),
    .height = @floatFromInt(self.swap_chain_extent.height),
    .minDepth = 0.0,
    .maxDepth = 1.0,
};
```

The scissor restricts rendering to a rectangular region:

```zig
const scissor = vk.VkRect2D{
    .offset = .{
        .x = 0,
        .y = 0,
    },
    .extent = self.swap_chain_extent,
};
```

The pipeline state points to those values:

```zig
const viewport_state = vk.VkPipelineViewportStateCreateInfo{
    .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_VIEWPORT_STATE_CREATE_INFO,
    .pNext = null,
    .flags = 0,
    .viewportCount = 1,
    .pViewports = &viewport,
    .scissorCount = 1,
    .pScissors = &scissor,
};
```

The values are also declared dynamic later. Vulkan reads these pointers during pipeline creation, so the local values must stay alive until that call returns.

### Configure rasterization

Rasterization converts primitives into fragments:

```zig
const rasterizer = vk.VkPipelineRasterizationStateCreateInfo{
    .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_RASTERIZATION_STATE_CREATE_INFO,
    .pNext = null,
    .flags = 0,
    .depthClampEnable = vk.VK_FALSE,
    .rasterizerDiscardEnable = vk.VK_FALSE,
    .polygonMode = vk.VK_POLYGON_MODE_FILL,
    .cullMode = vk.VK_CULL_MODE_BACK_BIT,
    .frontFace = vk.VK_FRONT_FACE_COUNTER_CLOCKWISE,
    .depthBiasEnable = vk.VK_FALSE,
    .depthBiasConstantFactor = 0.0,
    .depthBiasClamp = 0.0,
    .depthBiasSlopeFactor = 0.0,
    .lineWidth = 1.0,
};
```

The important choices are:

- `FILL`: fill triangle interiors rather than drawing only outlines,
- `BACK_BIT`: discard back-facing triangles,
- `COUNTER_CLOCKWISE`: treat counter-clockwise vertices as front-facing.

The front-face convention must match the coordinate system and vertex order used by the engine. If triangles disappear unexpectedly, face winding and culling are common causes.

### Configure multisampling

The initial renderer uses one sample per pixel:

```zig
const multisampling = vk.VkPipelineMultisampleStateCreateInfo{
    .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_MULTISAMPLE_STATE_CREATE_INFO,
    .pNext = null,
    .flags = 0,
    .rasterizationSamples = vk.VK_SAMPLE_COUNT_1_BIT,
    .sampleShadingEnable = vk.VK_FALSE,
    .minSampleShading = 1.0,
    .pSampleMask = null,
    .alphaToCoverageEnable = vk.VK_FALSE,
    .alphaToOneEnable = vk.VK_FALSE,
};
```

Multisampling can smooth polygon edges, but it increases memory use and requires compatible image attachments. Starting with one sample keeps the first pipeline small and predictable.

### Configure color blending

The fragment shader returns a color that should replace the destination pixel:

```zig
const color_blend_attachment = vk.VkPipelineColorBlendAttachmentState{
    .blendEnable = vk.VK_FALSE,
    .srcColorBlendFactor = vk.VK_BLEND_FACTOR_ONE,
    .dstColorBlendFactor = vk.VK_BLEND_FACTOR_ZERO,
    .colorBlendOp = vk.VK_BLEND_OP_ADD,
    .srcAlphaBlendFactor = vk.VK_BLEND_FACTOR_ONE,
    .dstAlphaBlendFactor = vk.VK_BLEND_FACTOR_ZERO,
    .alphaBlendOp = vk.VK_BLEND_OP_ADD,
    .colorWriteMask =
        vk.VK_COLOR_COMPONENT_R_BIT |
        vk.VK_COLOR_COMPONENT_G_BIT |
        vk.VK_COLOR_COMPONENT_B_BIT |
        vk.VK_COLOR_COMPONENT_A_BIT,
};
```

The attachment state is placed into the color-blend state:

```zig
const color_blending = vk.VkPipelineColorBlendStateCreateInfo{
    .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_COLOR_BLEND_STATE_CREATE_INFO,
    .pNext = null,
    .flags = 0,
    .logicOpEnable = vk.VK_FALSE,
    .logicOp = vk.VK_LOGIC_OP_COPY,
    .attachmentCount = 1,
    .pAttachments = &color_blend_attachment,
    .blendConstants = .{ 0.0, 0.0, 0.0, 0.0 },
};
```

Blending becomes important for transparent materials, particles, UI, and decals. It is disabled for this first opaque triangle.

### Configure dynamic state

Viewport and scissor values are marked dynamic:

```zig
const dynamic_states = [_]vk.VkDynamicState{
    vk.VK_DYNAMIC_STATE_VIEWPORT,
    vk.VK_DYNAMIC_STATE_SCISSOR,
};

const dynamic_state = vk.VkPipelineDynamicStateCreateInfo{
    .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_DYNAMIC_STATE_CREATE_INFO,
    .pNext = null,
    .flags = 0,
    .dynamicStateCount = @intCast(dynamic_states.len),
    .pDynamicStates = &dynamic_states,
};
```

The command buffer must set both values before drawing. The current lesson only creates the pipeline; command recording will use these dynamic states later.

### Build the dynamic-rendering structure

The rendering structure replaces the C++ `vk::PipelineRenderingCreateInfo` object:

```zig
var color_format = self.swap_chain_surface_format.format;

const rendering_info = vk.VkPipelineRenderingCreateInfo{
    .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_RENDERING_CREATE_INFO,
    .pNext = null,
    .viewMask = 0,
    .colorAttachmentCount = 1,
    .pColorAttachmentFormats = &color_format,
    .depthAttachmentFormat = vk.VK_FORMAT_UNDEFINED,
    .stencilAttachmentFormat = vk.VK_FORMAT_UNDEFINED,
};
```

This says that the pipeline will render to one color attachment using the swap-chain format. There is no depth or stencil attachment yet.

The device must support and enable dynamic rendering before this pipeline can be created. With Vulkan 1.4, dynamic rendering is part of the core API, but the corresponding device feature still needs to be enabled during logical-device creation.

```zig
        .subpass = 0,
        .basePipelineHandle = null,
        .basePipelineIndex = -1,
    };

    const result = vk.vkCreateGraphicsPipelines(
        self.device,
        null,
        1,
        &pipeline_create_info,
        null,
        &self.graphics_pipeline,
    );

    if (result != vk.VK_SUCCESS) {
        std.log.err(
            "Failed to create Vulkan graphics pipeline",
            .{},
        );
        return error.FailedToCreateGraphicsPipeline;
    }

    std.log.info("Created Vulkan graphics pipeline", .{});
}
```

This is the direct raw-binding replacement for:

```cpp
graphicsPipeline = vk::raii::Pipeline(
    device,
    nullptr,
    pipelineCreateInfoChain.get<vk::GraphicsPipelineCreateInfo>()
);
```

The important translation points are:

| C++ Vulkan-Hpp | Raw Zig binding |
|---|---|
| `vk::StructureChain` | Separate structures connected through `.pNext` |
| `vk::raii::Pipeline` | Nullable `vk.VkPipeline` field |
| RAII construction | `vk.vkCreateGraphicsPipelines` |
| `nullptr` handle | `null` |
| Hpp enum names | `vk.VK_*` constants |
| Hpp fields | C Vulkan fields such as `.sType`, `.pStages`, `.renderPass` |

### Destroy the graphics pipeline

Add a helper to the existing application:

```zig
fn destroyGraphicsPipeline(self: *HelloTriangleApplication) void {
    if (self.graphics_pipeline != null) {
        vk.vkDestroyPipeline(
            self.device,
            self.graphics_pipeline,
            null,
        );
        self.graphics_pipeline = null;
    }
}
```

The pipeline must be destroyed before its logical device.

### Update cleanup order

The graphics pipeline uses the pipeline layout and shader module, so destroy it first. The pipeline layout and shader module must also be destroyed before the logical device.

Place these calls near the beginning of the existing `cleanup()` method:

```zig
fn cleanup(self: *HelloTriangleApplication) void {
    self.destroyGraphicsPipeline();
    self.destroyPipelineLayout();

    if (self.shader_module != null) {
        vk.vkDestroyShaderModule(
            self.device,
            self.shader_module,
            null,
        );
        self.shader_module = null;
    }

    self.destroyImageViews();

    if (self.swap_chain != null) {
        vk.vkDestroySwapchainKHR(
            self.device,
            self.swap_chain,
            null,
        );
        self.swap_chain = null;
    }

    if (self.device != null) {
        vk.vkDestroyDevice(self.device, null);
        self.device = null;
    }

    if (self.surface != null) {
        vk.vkDestroySurfaceKHR(
            self.instance,
            self.surface,
            null,
        );
        self.surface = null;
    }

    // Keep the existing debug-messenger cleanup here.

    if (self.instance != null) {
        vk.vkDestroyInstance(self.instance, null);
        self.instance = null;
    }

    if (self.window) |window| {
        sdl.SDL_DestroyWindow(window);
        self.window = null;
    }

    sdl.SDL_Quit();
}
```

The complete dependency order is:

```text
graphics pipeline
pipeline layout
shader module
image views
swap chain
logical device
surface
debug messenger
instance
SDL window
SDL
```

The exact order of the pipeline layout and shader module is not as important as destroying both after the graphics pipeline and before the logical device. The pipeline must not outlive either object it uses.

### Important compatibility checks

If pipeline creation fails, check these areas first:

1. **Dynamic rendering feature**

   The logical device must enable dynamic rendering. Vulkan 1.4 provides the feature in the core API, but features still need to be enabled during device creation.

2. **Shader entry points**

   The pipeline uses:

   ```zig
   .pName = "vertMain"
   .pName = "fragMain"
   ```

   These names must exist in `shaders/slang.spv`.

3. **Swap-chain format**

   The dynamic-rendering format must be the same format selected during swap-chain creation:

   ```zig
   self.swap_chain_surface_format.format
   ```

4. **Pipeline layout**

   `self.pipeline_layout` must contain a valid Vulkan pipeline-layout handle.

5. **Viewport extent**

   The swap-chain extent must be valid and non-zero. A minimized window may not have a drawable extent. Swap-chain recreation and suspension handling will be addressed in a later lesson.

6. **Pointer lifetime**

   All structures passed to `vk.vkCreateGraphicsPipelines` are local variables, but they remain alive until that function returns. Do not return the pipeline-create structure from a helper that also returns pointers to temporary local values.

## Recap & What's Next

The C++ pipeline chain:

```cpp
vk::StructureChain<
    vk::GraphicsPipelineCreateInfo,
    vk::PipelineRenderingCreateInfo
>
```

became two raw Vulkan structures connected explicitly:

```zig
const rendering_info = vk.VkPipelineRenderingCreateInfo{
    .sType = vk.VK_STRUCTURE_TYPE_PIPELINE_RENDERING_CREATE_INFO,
    // ...
};

const pipeline_create_info = vk.VkGraphicsPipelineCreateInfo{
    .sType = vk.VK_STRUCTURE_TYPE_GRAPHICS_PIPELINE_CREATE_INFO,
    .pNext = &rendering_info,
    // ...
};
```

The RAII pipeline:

```cpp
vk::raii::Pipeline graphicsPipeline = nullptr;
```

became an explicitly owned nullable handle:

```zig
graphics_pipeline: vk.VkPipeline = null,
```

and creation uses the raw Vulkan function:

```zig
const result = vk.vkCreateGraphicsPipelines(
    self.device,
    null,
    1,
    &pipeline_create_info,
    null,
    &self.graphics_pipeline,
);

if (result != vk.VK_SUCCESS) {
    return error.FailedToCreateGraphicsPipeline;
}
```

The important lessons are:

- Vulkan's `pNext` chains are ordinary linked structures at the C API level.
- Dynamic rendering supplies the swap-chain format through `VkPipelineRenderingCreateInfo`.
- A graphics pipeline combines shader stages and fixed-function state.
- The first pipeline can use an empty vertex-input description because the shaders use `SV_VertexID`.
- Nullable Vulkan handles make ownership and cleanup explicit.
- Pipeline creation errors should be returned as named Zig errors.
- The graphics pipeline must be destroyed before the pipeline layout, shader module, image views, swap chain, and logical device.

Koba now has a pipeline description capable of rendering a triangle. The next step is to create command pools and command buffers, set the dynamic viewport and scissor, begin dynamic rendering, bind the pipeline, issue a draw command, and present the rendered swap-chain image.
