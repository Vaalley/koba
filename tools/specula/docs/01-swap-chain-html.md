# Swap Chain :: Vulkan Documentation Project (Zig 0.16.0 + SDL3)

## Overview

The previous lessons prepared the pieces needed to display rendered images:

1. Koba creates an SDL3 window.
2. Vulkan creates an instance.
3. Vulkan creates a window surface through SDL3.
4. Koba selects a physical device.
5. Koba creates a logical device and graphics queue.

This lesson connects those pieces by creating a **swap chain**.

A swap chain is a group of images owned by Vulkan and associated with the window
surface. Rendering does not normally draw directly into the window. Instead, the
engine renders into one swap-chain image, then presents that image to the
window.

The swap-chain setup has five important stages:

1. Ask the physical device what the surface supports.
2. Choose an image format.
3. Choose a presentation mode.
4. Choose the swap-chain extent, or size.
5. Create the swap chain and retrieve its images.

The C++ tutorial uses `vk::raii::SwapchainKHR`, vector-returning helper
functions, and Vulkan-Hpp enum names. Koba uses raw C bindings generated by
`addTranslateC`, so the translation must call functions such as:

```zig
vk.vkGetPhysicalDeviceSurfaceCapabilitiesKHR(...)
vk.vkCreateSwapchainKHR(...)
vk.vkGetSwapchainImagesKHR(...)
```

There are no `SwapchainProxy` objects and no methods such as
`device.createSwapchainKHR`.

This lesson extends the existing `HelloTriangleApplication` in `src/main.zig`.
It does not introduce a second application type or a new module.

## Concepts & Explanations

### Why a swap chain is necessary

A window is controlled by the operating system and window system. Vulkan cannot
simply assume that the window accepts any image format, size, or presentation
strategy.

The surface reports the constraints imposed by the platform and presentation
engine. Those constraints include:

- which pixel formats are available,
- which presentation modes are available,
- the minimum and maximum number of images,
- the supported image dimensions,
- supported transformations,
- supported alpha-compositing modes.

The swap chain must be created using values that satisfy those constraints.

A useful mental model is:

```text
physical device + surface
        |
        v
surface support details
        |
        v
chosen format, mode, extent, image count
        |
        v
swap chain images
```

The swap-chain images will later become the targets used by image views,
framebuffers, and render passes.

### Raw C bindings change the call style

The previous lessons may have shown wrapper-style code. Koba's actual bindings
are raw translated C declarations.

For example, this is the required style:

```zig
const result = vk.vkCreateSwapchainKHR(
    self.device,
    &create_info,
    null,
    &self.swap_chain,
);
if (result != vk.VK_SUCCESS) {
    return error.FailedToCreateSwapchain;
}
```

Do not translate this into:

```zig
self.device.createSwapchainKHR(...)
```

and do not use:

```zig
vk.DeviceProxy
vk.SwapchainProxy
vk.InstanceProxy
```

The raw binding makes Vulkan's pointer-and-count API visible. That is especially
important for querying lists such as surface formats, presentation modes, and
swap-chain images.

### Vulkan's two-call query pattern

Many Vulkan C functions return a variable-length list. They use this pattern:

1. Pass a pointer to a count and a null data pointer.
2. Allocate enough memory for that count.
3. Call the function again with the allocated buffer.

For example:

```zig
var format_count: u32 = 0;

var result = vk.vkGetPhysicalDeviceSurfaceFormatsKHR(
    self.physical_device,
    self.surface,
    &format_count,
    null,
);
if (result != vk.VK_SUCCESS) {
    return error.FailedToQuerySurfaceFormats;
}

const formats = try self.allocator.alloc(vk.VkSurfaceFormatKHR, format_count);
defer self.allocator.free(formats);

result = vk.vkGetPhysicalDeviceSurfaceFormatsKHR(
    self.physical_device,
    self.surface,
    &format_count,
    formats.ptr,
);
if (result != vk.VK_SUCCESS) {
    return error.FailedToQuerySurfaceFormats;
}
```

The temporary query buffer is freed immediately after choosing a value. The
actual swap-chain images are different: those handles are needed later, so they
remain stored in the application.

### Surface formats

A `VkSurfaceFormatKHR` contains:

- `.format`: the pixel layout, such as BGRA with 8 bits per channel,
- `.colorSpace`: how color values are interpreted.

The tutorial prefers:

```zig
vk.VK_FORMAT_B8G8R8A8_SRGB
vk.VK_COLOR_SPACE_SRGB_NONLINEAR_KHR
```

That is a preference, not a universal requirement. Drivers may not provide that
exact combination. The safe strategy is:

1. Search for the preferred combination.
2. Fall back to the first available format.

The fallback matters because a perfectly valid Vulkan device is allowed to
expose a different format list.

### Presentation modes

The presentation mode controls how completed images are delivered to the
display.

Common modes include:

- `VK_PRESENT_MODE_FIFO_KHR`: waits for the display's vertical refresh and
  behaves like vertical synchronization.
- `VK_PRESENT_MODE_MAILBOX_KHR`: replaces queued images with newer ones and can
  reduce visible tearing while allowing low latency.
- `VK_PRESENT_MODE_IMMEDIATE_KHR`: presents immediately and may tear.

`FIFO` is guaranteed by the Vulkan specification, so it is a reliable fallback.
`MAILBOX` is a useful preference when available, but it is not guaranteed.

For a first engine, preferring mailbox and falling back to FIFO is a reasonable
trade-off:

- mailbox can provide smoother interactive rendering,
- FIFO is universally available,
- immediate mode may produce tearing.

### Swap-chain extent and high-DPI windows

The extent is the pixel size of the swap-chain images.

A common platform distinction is:

- some platforms provide a fixed `currentExtent`,
- other platforms use the special value `UINT32_MAX` to indicate that the
  application should choose the size.

SDL3's `SDL_GetWindowSizeInPixels` is important here. `SDL_GetWindowSize` may
return logical window units, while Vulkan images must use actual drawable
pixels. This distinction matters on high-DPI displays.

The selected size must be clamped between:

```zig
capabilities.minImageExtent
capabilities.maxImageExtent
```

Do not convert a negative window size directly to `u32`. A minimized or
otherwise unavailable window can report a size that is not usable for swap-chain
creation. A later resize lesson will handle suspension and recreation more
fully.

### Image count

The minimum image count is a lower bound, not always the best count.

The usual tutorial choice is:

```zig
surface_capabilities.minImageCount + 1
```

This gives the application one extra image so that rendering can continue while
another image is waiting for presentation.

The count must still respect `.maxImageCount`. A value of zero means that the
surface has no upper limit.

The helper below chooses at least three images, as in the supplied source, and
then clamps to the maximum when one exists. The trade-off is that more images
can improve throughput but also increase latency and memory use.

### Image usage and sharing mode

The swap-chain creation structure says how the images will be used.

For the current renderer:

```zig
.imageUsage = vk.VK_IMAGE_USAGE_COLOR_ATTACHMENT_BIT
```

means that the images will be used as color-rendering targets.

The engine uses one queue family for graphics and presentation, so:

```zig
.imageSharingMode = vk.VK_SHARING_MODE_EXCLUSIVE
```

is appropriate. Exclusive ownership usually has the least overhead.

If graphics and presentation later use different queue families, the code will
need either:

- concurrent sharing mode with both family indices, or
- explicit queue-family ownership transfers.

### Cleanup order

The swap chain depends on the logical device. Therefore it must be destroyed
before the device:

```text
swap chain
logical device
surface
debug messenger
instance
SDL window
SDL
```

The swap-chain images themselves are owned by the swap chain. Koba stores their
handles but must not call `vkDestroyImage` on them. Destroying the swap chain
releases those images.

## Code Translation Sections

### Add swap-chain state to `HelloTriangleApplication`

Add these fields to the existing struct in `src/main.zig`:

```zig
swap_chain: vk.VkSwapchainKHR = null,
swap_chain_images: []vk.VkImage = &.{},
swap_chain_surface_format: vk.VkSurfaceFormatKHR = undefined,
swap_chain_extent: vk.VkExtent2D = undefined,
```

The empty slice is only a starting value. Once `createSwapChain` succeeds, it
owns memory allocated through `self.allocator`.

Because the images are stored for later lessons, they are not freed at the end
of `createSwapChain`. They are freed in `cleanup()`.

### Define the required device extension

The swap-chain extension must already be enabled when the logical device is
created.

Keep the extension list at module scope and reuse it in `VkDeviceCreateInfo`:

```zig
const required_device_extensions = [_][*:0]const u8{
    "VK_KHR_swapchain",
};
```

The logical-device creation code should contain the equivalent of:

```zig
.enabledExtensionCount = @intCast(required_device_extensions.len),
.ppEnabledExtensionNames = &required_device_extensions,
```

The important point is that the extension is enabled on the **logical device**,
not only on the Vulkan instance.

### Add a small Vulkan result checker

The existing project uses explicit `VkResult` checks. A helper can keep the
repeated checks readable while preserving that style:

```zig
fn checkVkResult(result: vk.VkResult, failure: anyerror) !void {
    if (result != vk.VK_SUCCESS) {
        std.log.err("Vulkan call failed with VkResult {d}", .{@intFromEnum(result)});
        return failure;
    }
}
```

If the generated binding represents `VkResult` as an integer rather than an
enum, use the project's existing logging and comparison form instead. The
important rule is that every result-returning Vulkan call is checked explicitly.

### Query surface capabilities

`VkSurfaceCapabilitiesKHR` describes the limits for the swap chain:

```zig
fn querySurfaceCapabilities(self: *HelloTriangleApplication) !vk.VkSurfaceCapabilitiesKHR {
    var capabilities: vk.VkSurfaceCapabilitiesKHR = undefined;

    const result = vk.vkGetPhysicalDeviceSurfaceCapabilitiesKHR(
        self.physical_device,
        self.surface,
        &capabilities,
    );

    if (result != vk.VK_SUCCESS) {
        return error.FailedToQuerySurfaceCapabilities;
    }

    return capabilities;
}
```

This call fills one structure directly, so it does not use the two-call list
pattern.

### Query surface formats

```zig
fn querySurfaceFormats(
    self: *HelloTriangleApplication,
) ![]vk.VkSurfaceFormatKHR {
    var format_count: u32 = 0;

    var result = vk.vkGetPhysicalDeviceSurfaceFormatsKHR(
        self.physical_device,
        self.surface,
        &format_count,
        null,
    );
    if (result != vk.VK_SUCCESS) {
        return error.FailedToQuerySurfaceFormats;
    }

    if (format_count == 0) {
        return error.NoSurfaceFormats;
    }

    const formats = try self.allocator.alloc(
        vk.VkSurfaceFormatKHR,
        format_count,
    );
    errdefer self.allocator.free(formats);

    result = vk.vkGetPhysicalDeviceSurfaceFormatsKHR(
        self.physical_device,
        self.surface,
        &format_count,
        formats.ptr,
    );
    if (result != vk.VK_SUCCESS) {
        return error.FailedToQuerySurfaceFormats;
    }

    return formats;
}
```

The caller owns the returned allocation and must free it. Returning an allocated
slice is useful here because the caller needs to inspect it while choosing the
format.

failures explicit and easier to diagnose.

### Choose a surface format

The C++ `std::ranges::find_if` becomes a Zig loop:

```zig
fn chooseSwapSurfaceFormat(
    available_formats: []const vk.VkSurfaceFormatKHR,
) vk.VkSurfaceFormatKHR {
    for (available_formats) |format| {
        if (format.format == vk.VK_FORMAT_B8G8R8A8_SRGB and
            format.colorSpace == vk.VK_COLOR_SPACE_SRGB_NONLINEAR_KHR)
        {
            return format;
        }
    }

    return available_formats[0];
}
```

The caller must ensure that `available_formats.len > 0` before calling this
function. The query function above already enforces that condition.

The fields intentionally use Vulkan's C names:

```zig
.format
.colorSpace
```

Do not change them to `.image_format` or `.color_space`.

### Choose a presentation mode

```zig
fn chooseSwapPresentMode(
    available_modes: []const vk.VkPresentModeKHR,
) !vk.VkPresentModeKHR {
    var has_fifo = false;

    for (available_modes) |mode| {
        if (mode == vk.VK_PRESENT_MODE_MAILBOX_KHR) {
            return vk.VK_PRESENT_MODE_MAILBOX_KHR;
        }

        if (mode == vk.VK_PRESENT_MODE_FIFO_KHR) {
            has_fifo = true;
        }
    }

    if (has_fifo) {
        return vk.VK_PRESENT_MODE_FIFO_KHR;
    }

    return error.NoFifoPresentMode;
}
```

The `has_fifo` check makes the specification requirement visible instead of
silently assuming it. If the driver violates the expected contract, the engine
reports a useful error.

### Choose the minimum image count

```zig
fn chooseSwapMinImageCount(
    capabilities: vk.VkSurfaceCapabilitiesKHR,
) u32 {
    var image_count = @max(3, capabilities.minImageCount);

    if (capabilities.maxImageCount != 0 and
        image_count > capabilities.maxImageCount)
    {
        image_count = capabilities.maxImageCount;
    }

    return image_count;
}
```

The `maxImageCount == 0` case means “no maximum.” It must not be treated as a
maximum of zero, or every requested image count would be invalid.

### Choose the swap-chain extent

```zig
fn chooseSwapExtent(
    self: *HelloTriangleApplication,
    capabilities: vk.VkSurfaceCapabilitiesKHR,
) !vk.VkExtent2D {
    if (capabilities.currentExtent.width != std.math.maxInt(u32)) {
        return capabilities.currentExtent;
    }

    var width: c_int = 0;
    var height: c_int = 0;

    const window = self.window orelse return error.WindowNotCreated;

    if (!sdl.SDL_GetWindowSizeInPixels(window, &width, &height)) {
        std.log.err("SDL_GetWindowSizeInPixels failed: {s}", .{
            sdl.SDL_GetError(),
        });
        return error.FailedToGetDrawableSize;
    }

    if (width <= 0 or height <= 0) {
        return error.WindowHasNoDrawableSize;
    }

    const pixel_width: u32 = @intCast(width);
    const pixel_height: u32 = @intCast(height);

    return .{
        .width = std.math.clamp(
            pixel_width,
            capabilities.minImageExtent.width,
            capabilities.maxImageExtent.width,
        ),
        .height = std.math.clamp(
            pixel_height,
            capabilities.minImageExtent.height,
            capabilities.maxImageExtent.height,
        ),
    };
}
```

`SDL_GetWindowSizeInPixels` is used instead of the logical window-size function
because Vulkan needs the actual framebuffer pixel dimensions.

A minimized window may have a zero drawable size. This code returns an error
rather than creating an invalid extent. During swap-chain recreation, the engine
will eventually need to wait until the window has a nonzero size.

### Create the swap chain

Add this method to the existing `HelloTriangleApplication` struct:

```zig
fn createSwapChain(self: *HelloTriangleApplication) !void {
    const capabilities = try self.querySurfaceCapabilities();

    const formats = try self.querySurfaceFormats();
    defer self.allocator.free(formats);

    const present_modes = try self.queryPresentModes();
    defer self.allocator.free(present_modes);

    self.swap_chain_surface_format = chooseSwapSurfaceFormat(formats);

    const present_mode = try chooseSwapPresentMode(present_modes);
    self.swap_chain_extent = try self.chooseSwapExtent(capabilities);

    const image_count = chooseSwapMinImageCount(capabilities);

    const create_info = vk.VkSwapchainCreateInfoKHR{
        .sType = vk.VK_STRUCTURE_TYPE_SWAPCHAIN_CREATE_INFO_KHR,
        .pNext = null,
        .flags = 0,
        .surface = self.surface,
        .minImageCount = image_count,
        .imageFormat = self.swap_chain_surface_format.format,
        .imageColorSpace = self.swap_chain_surface_format.colorSpace,
        .imageExtent = self.swap_chain_extent,
        .imageArrayLayers = 1,
        .imageUsage = vk.VK_IMAGE_USAGE_COLOR_ATTACHMENT_BIT,
        .imageSharingMode = vk.VK_SHARING_MODE_EXCLUSIVE,
        .queueFamilyIndexCount = 0,
        .pQueueFamilyIndices = null,
        .preTransform = capabilities.currentTransform,
        .compositeAlpha = vk.VK_COMPOSITE_ALPHA_OPAQUE_BIT_KHR,
        .presentMode = present_mode,
        .clipped = vk.VK_TRUE,
        .oldSwapchain = null,
    };

    var result = vk.vkCreateSwapchainKHR(
        self.device,
        &create_info,
        null,
        &self.swap_chain,
    );
    if (result != vk.VK_SUCCESS) {
        return error.FailedToCreateSwapchain;
    }

    errdefer {
        vk.vkDestroySwapchainKHR(self.device, self.swap_chain, null);
        self.swap_chain = null;
    }

    var actual_image_count: u32 = 0;

    result = vk.vkGetSwapchainImagesKHR(
        self.device,
        self.swap_chain,
        &actual_image_count,
        null,
    );
    if (result != vk.VK_SUCCESS) {
        return error.FailedToGetSwapchainImages;
    }

    if (actual_image_count == 0) {
        return error.NoSwapchainImages;
    }

    self.swap_chain_images = try self.allocator.alloc(
        vk.VkImage,
        actual_image_count,
    );

    errdefer {
        self.allocator.free(self.swap_chain_images);
        self.swap_chain_images = &.{};
    }

    result = vk.vkGetSwapchainImagesKHR(
        self.device,
        self.swap_chain,
        &actual_image_count,
        self.swap_chain_images.ptr,
    );
    if (result != vk.VK_SUCCESS) {
        return error.FailedToGetSwapchainImages;
    }

    std.log.info(
        "Created swap chain with {d} images at {d}x{d}",
        .{
            self.swap_chain_images.len,
            self.swap_chain_extent.width,
            self.swap_chain_extent.height,
        },
    );
}
```

The `oldSwapchain` field is null during initial creation. It becomes useful
during window resizing: the old swap chain can be supplied while creating a
replacement, allowing Vulkan to retire resources in an orderly way.

The `errdefer` blocks are important because creation has multiple failure
points. If image retrieval fails after the swap chain was created, the partially
initialized Vulkan object must still be destroyed.

### Retrieve swap-chain images

The image handles returned by `vkGetSwapchainImagesKHR` are owned by the swap
chain. Koba stores them for future image-view creation:

```zig
swap_chain_images: []vk.VkImage = &.{},
```

Do not destroy them individually:

```zig
// Incorrect:
vk.vkDestroyImage(self.device, image, null);
```

The correct owner is the swap chain:

```zig
vk.vkDestroySwapchainKHR(self.device, self.swap_chain, null);
```

This ownership distinction will become important again when image views and
framebuffers are added.

### Destroy the swap chain during cleanup

Extend the existing `cleanup()` method. The swap chain must be destroyed before
the logical device:

```zig
fn cleanup(self: *HelloTriangleApplication) void {
    if (self.swap_chain_images.len != 0) {
        self.allocator.free(self.swap_chain_images);
        self.swap_chain_images = &.{};
    }

    if (self.swap_chain != null and self.device != null) {
        vk.vkDestroySwapchainKHR(
            self.device,
            self.swap_chain,
            null,
        );
        self.swap_chain = null;
    }

    if (self.device != null) {
        vk.vkDestroyDevice(self.device, null);
        self.device = null;
    }

    if (self.surface != null and self.instance != null) {
        vk.vkDestroySurfaceKHR(
            self.instance,
            self.surface,
            null,
        );
        self.surface = null;
    }

    // Keep the remaining existing cleanup order:
    // debug messenger
    // Vulkan instance
    // SDL window
    // SDL_Quit
}
```

If the existing project uses a different conditional style for optional Vulkan
handles, keep that style. The required lifetime order is the important part:

```text
free CPU-side image slice
destroy VkSwapchainKHR
destroy VkDevice
destroy VkSurfaceKHR
destroy debug messenger
destroy VkInstance
destroy SDL_Window
SDL_Quit
```

```zig
            self.physical_device,
            self.surface,
            &mode_count,
            null,
        );
        if (result != vk.VK_SUCCESS) {
            return error.FailedToQueryPresentModes;
        }

        if (mode_count == 0) {
            return error.NoPresentModes;
        }

        const modes = try self.allocator.alloc(
            vk.VkPresentModeKHR,
            mode_count,
        );
        errdefer self.allocator.free(modes);

        result = vk.vkGetPhysicalDeviceSurfacePresentModesKHR(
            self.physical_device,
            self.surface,
            &mode_count,
            modes.ptr,
        );
        if (result != vk.VK_SUCCESS) {
            return error.FailedToQueryPresentModes;
        }

        return modes;
    }

    fn chooseSwapExtent(
        self: *HelloTriangleApplication,
        capabilities: vk.VkSurfaceCapabilitiesKHR,
    ) !vk.VkExtent2D {
        if (capabilities.currentExtent.width != std.math.maxInt(u32)) {
            return capabilities.currentExtent;
        }

        const window = self.window orelse return error.WindowNotCreated;

        var width: c_int = 0;
        var height: c_int = 0;

        if (!sdl.SDL_GetWindowSizeInPixels(window, &width, &height)) {
            std.log.err("SDL_GetWindowSizeInPixels failed: {s}", .{
                sdl.SDL_GetError(),
            });
            return error.FailedToGetDrawableSize;
        }

        if (width <= 0 or height <= 0) {
            return error.WindowHasNoDrawableSize;
        }

        const pixel_width: u32 = @intCast(width);
        const pixel_height: u32 = @intCast(height);

        return .{
            .width = std.math.clamp(
                pixel_width,
                capabilities.minImageExtent.width,
                capabilities.maxImageExtent.width,
            ),
            .height = std.math.clamp(
                pixel_height,
                capabilities.minImageExtent.height,
                capabilities.maxImageExtent.height,
            ),
        };
    }

    fn createSwapChain(
        self: *HelloTriangleApplication,
    ) !void {
        const capabilities = try self.querySurfaceCapabilities();

        const formats = try self.querySurfaceFormats();
        defer self.allocator.free(formats);

        const present_modes = try self.queryPresentModes();
        defer self.allocator.free(present_modes);

        self.swap_chain_surface_format =
            chooseSwapSurfaceFormat(formats);

        const present_mode =
            try chooseSwapPresentMode(present_modes);

        self.swap_chain_extent =
            try self.chooseSwapExtent(capabilities);

        const image_count =
            chooseSwapMinImageCount(capabilities);

        const create_info = vk.VkSwapchainCreateInfoKHR{
            .sType = vk.VK_STRUCTURE_TYPE_SWAPCHAIN_CREATE_INFO_KHR,
            .pNext = null,
            .flags = 0,
            .surface = self.surface,
            .minImageCount = image_count,
            .imageFormat = self.swap_chain_surface_format.format,
            .imageColorSpace =
                self.swap_chain_surface_format.colorSpace,
            .imageExtent = self.swap_chain_extent,
            .imageArrayLayers = 1,
            .imageUsage =
                vk.VK_IMAGE_USAGE_COLOR_ATTACHMENT_BIT,
            .imageSharingMode =
                vk.VK_SHARING_MODE_EXCLUSIVE,
            .queueFamilyIndexCount = 0,
            .pQueueFamilyIndices = null,
            .preTransform = capabilities.currentTransform,
            .compositeAlpha =
                vk.VK_COMPOSITE_ALPHA_OPAQUE_BIT_KHR,
            .presentMode = present_mode,
            .clipped = vk.VK_TRUE,
            .oldSwapchain = null,
        };

        var result = vk.vkCreateSwapchainKHR(
            self.device,
            &create_info,
            null,
            &self.swap_chain,
        );
        if (result != vk.VK_SUCCESS) {
            return error.FailedToCreateSwapchain;
        }

        errdefer {
            vk.vkDestroySwapchainKHR(
                self.device,
                self.swap_chain,
                null,
            );
            self.swap_chain = null;
        }

        var actual_image_count: u32 = 0;

        result = vk.vkGetSwapchainImagesKHR(
            self.device,
            self.swap_chain,
            &actual_image_count,
            null,
        );
        if (result != vk.VK_SUCCESS) {
            return error.FailedToGetSwapchainImages;
        }

        if (actual_image_count == 0) {
            return error.NoSwapchainImages;
        }

        self.swap_chain_images = try self.allocator.alloc(
            vk.VkImage,
            actual_image_count,
        );

        errdefer {
            self.allocator.free(self.swap_chain_images);
            self.swap_chain_images = &.{};
        }

        result = vk.vkGetSwapchainImagesKHR(
            self.device,
            self.swap_chain,
            &actual_image_count,
            self.swap_chain_images.ptr,
        );
        if (result != vk.VK_SUCCESS) {
            return error.FailedToGetSwapchainImages;
        }

        std.log.info(
            "Created swap chain with {d} images at {d}x{d}",
            .{
                self.swap_chain_images.len,
                self.swap_chain_extent.width,
                self.swap_chain_extent.height,
            },
        );
    }

    fn cleanupSwapChain(
        self: *HelloTriangleApplication,
    ) void {
        if (self.swap_chain_images.len != 0) {
            self.allocator.free(self.swap_chain_images);
            self.swap_chain_images = &.{};
        }

        if (self.swap_chain != null) {
            vk.vkDestroySwapchainKHR(
                self.device,
                self.swap_chain,
                null,
            );
            self.swap_chain = null;
        }
    }
};
```

## Recap & What's Next

The swap chain is the first object that turns surface capabilities into a
renderable presentation pipeline.

You learned how to:

- enable `VK_KHR_swapchain` on the logical device,
- query surface capabilities,
- query variable-length format and presentation-mode lists,
- use allocator-backed slices for temporary Vulkan results,
- choose a preferred format with a safe fallback,
- prefer mailbox presentation while retaining FIFO compatibility,
- clamp the drawable size to surface limits,
- create a `VkSwapchainKHR` with raw Vulkan C calls,
- retrieve and store swap-chain image handles,
- destroy the swap chain before destroying the logical device.

The main ownership rule is:

```text
Koba allocates the CPU-side image slice.
The swap chain owns the Vulkan swap-chain images.
The device owns the swap-chain object.
```

The next rendering step is to create an **image view** for every swap-chain
image. Image views describe how shaders and render passes access images. After
that, the engine can create render passes or dynamic-rendering state,
framebuffers, synchronization objects, and finally the draw-and-present loop.

Window resizing will also require swap-chain recreation. When the drawable size
changes, Koba must wait for a valid nonzero SDL pixel size, destroy the old swap
chain and its dependent objects, then build a new one using the updated surface
capabilities.
