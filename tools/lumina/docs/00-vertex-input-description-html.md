# Vertex Input Description in Vulkan and Zig

## Overview

This lesson connects the vertex shader’s inputs to the data that Vulkan will eventually read from a vertex buffer. The current koba shader generates vertices from `SV_VertexID`, so it does not yet need vertex input descriptions. We will change it to consume position and color attributes, define the matching Zig vertex type, and configure the graphics pipeline with Vulkan’s raw C bindings. The actual GPU vertex buffer will be created in the next step; this lesson establishes the layout contract that buffer must follow.

## Concepts & Explanations

### The vertex input contract

A vertex shader can receive data in several ways. The current shader uses:

```slang
SV_VertexID
```

That is a built-in integer generated by Vulkan for each draw call. No vertex buffer is required.

The lesson changes the shader to receive:

- location `0`: a two-component position
- location `1`: a three-component color

The CPU-side vertex structure must therefore have the same logical layout:

```text
location 0 -> position: float2
location 1 -> color:    float3
```

Vulkan does not inspect a C++ or Zig struct and infer this relationship. The pipeline must explicitly describe:

1. **Bindings** — where vertex records come from and how large each record is.
2. **Attributes** — which fields within each record correspond to shader locations.

This separation is useful when a pipeline uses multiple buffers. For example, positions could come from binding `0` and colors from binding `1`. In this lesson both attributes are interleaved in one buffer.

### Translating `glm::vec2` and `glm::vec3`

The C++ lesson uses:

```cpp
glm::vec2 pos;
glm::vec3 color;
```

For this simple vertex format, two and three `f32` values are sufficient in Zig:

```zig
pos: [2]f32,
color: [3]f32,
```

The vertex type should be an `extern struct`:

```zig
const Vertex = extern struct {
    pos: [2]f32,
    color: [3]f32,
};
```

`extern struct` is important because it requests C-compatible field layout. It makes the layout suitable for communicating with an API that consumes raw memory, such as Vulkan.

For this type:

- `pos` begins at byte offset `0`
- `color` begins at byte offset `8`
- the total stride is `20` bytes

The offsets should still be calculated rather than hard-coded. That keeps the Vulkan description correct if the struct changes later.

### Binding descriptions

A binding description describes one stream of vertex records:

```zig
binding = 0
stride = @sizeOf(Vertex)
inputRate = VK_VERTEX_INPUT_RATE_VERTEX
```

`VK_VERTEX_INPUT_RATE_VERTEX` means Vulkan advances to the next record for every vertex. The alternative, `VK_VERTEX_INPUT_RATE_INSTANCE`, advances once per instance and is useful for instanced rendering.

The stride is the distance from one vertex record to the next. With the `Vertex` type above, it is `20` bytes.

### Attribute descriptions

An attribute description maps one shader input location to bytes in a binding:

```text
location 0 -> binding 0, R32G32_SFLOAT, offset of pos
location 1 -> binding 0, R32G32B32_SFLOAT, offset of color
```

The Vulkan formats are not descriptions of Zig types in general. They describe exactly how the shader interprets the bytes:

- `VK_FORMAT_R32G32_SFLOAT` is two 32-bit floating-point values.
- `VK_FORMAT_R32G32B32_SFLOAT` is three 32-bit floating-point values.

The format must match the shader input. Supplying a three-component format for a two-component shader input, or using integer formats for floating-point values, produces incorrect data or validation errors.

### Why the descriptions belong in the pipeline

The vertex input state is part of the graphics pipeline. This is one of Vulkan’s central design choices: pipeline state is made explicit and mostly immutable.

The benefit is predictable GPU state and efficient command recording. The trade-off is that changing a vertex layout generally requires another graphics pipeline. For koba, this is appropriate because different material or mesh layouts can later have their own pipeline configurations.

### Shader locations and Slang

The shader must use the same locations as the Vulkan attribute descriptions. With Slang, the input fields are assigned locations according to the shader interface. To make the contract explicit, use location annotations where supported by the Slang version used by the project:

```slang
struct VSInput {
    [[vk::location(0)]] float2 inPosition;
    [[vk::location(1)]] float3 inColor;
};
```

This avoids relying on compiler-assigned locations. The important rule is that the location numbers and Vulkan attribute descriptions must agree.

### A subtle transition in koba

After this lesson, the shader expects vertex-buffer data. The existing draw path still uses the old `SV_VertexID` shader and does not bind a vertex buffer. Therefore, the complete vertex-buffer rendering path is not finished yet.

The correct progression is:

1. Define the CPU vertex type.
2. Describe its layout to the pipeline.
3. Create and upload a Vulkan vertex buffer.
4. Bind that buffer before `vkCmdDraw`.
5. Draw the vertices.

This lesson implements steps 1 and 2 and updates the shader for step 3.

## Code Translation

### 1. Replace the shader

Replace `shaders/shader.slang` with:

```zig
```

Shader files are Slang rather than Zig, so the code itself remains Slang:

```slang
struct VSInput {
    [[vk::location(0)]] float2 inPosition;
    [[vk::location(1)]] float3 inColor;
};

struct VSOutput {
    float4 pos : SV_Position;
    float3 color;
};

[shader("vertex")]
VSOutput vertMain(VSInput input) {
    VSOutput output;
    output.pos = float4(input.inPosition, 0.0, 1.0);
    output.color = input.inColor;
    return output;
}

[shader("fragment")]
float4 fragMain(VSOutput vertIn) : SV_TARGET {
    return float4(vertIn.color, 1.0);
}
```

The `[[vk::location(...)]]` annotations make the Vulkan interface explicit. Recompile the shader using the project’s existing command:

```text
slangc shaders/shader.slang -target spirv -profile spirv_1_4 -emit-spirv-directly -fvk-use-entrypoint-name -entry vertMain -entry fragMain -o shaders/slang.spv
```

The generated SPIR-V must be available at the path already used by `main.zig`.

### 2. Add the Zig vertex type and sample data

Add these declarations at file scope in `src/main.zig`, near the other constants:

```zig
const Vertex = extern struct {
    pos: [2]f32,
    color: [3]f32,
};

const vertices = [_]Vertex{
    .{
        .pos = .{ 0.0, -0.5 },
        .color = .{ 1.0, 0.0, 0.0 },
    },
    .{
        .pos = .{ 0.5, 0.5 },
        .color = .{ 0.0, 1.0, 0.0 },
    },
    .{
        .pos = .{ -0.5, 0.5 },
        .color = .{ 0.0, 0.0, 1.0 },
    },
};
```

The array is the Zig equivalent of the C++ `std::vector<Vertex>` initializer. It is currently CPU-side data only. The next lesson will use it when creating the Vulkan vertex buffer.

Keeping the data as an array is useful here because:

- its length is known at compile time;
- `vertices.len` can later become the vertex count;
- `&vertices` or `vertices[0..].ptr` can be passed to an upload operation.

### 3. Add vertex description helpers

Add these free functions in the “Vulkan Helpers” section at the bottom of `main.zig`. They do not need an `Application` instance, so they should not be methods with an unused `self` parameter.

```zig
fn vertex_binding_description() vulkan.VkVertexInputBindingDescription {
    return .{
        .binding = 0,
        .stride = @intCast(@sizeOf(Vertex)),
        .inputRate = vulkan.VK_VERTEX_INPUT_RATE_VERTEX,
    };
}

fn vertex_attribute_descriptions() [2]vulkan.VkVertexInputAttributeDescription {
    return .{
        .{
            .location = 0,
            .binding = 0,
            .format = vulkan.VK_FORMAT_R32G32_SFLOAT,
            .offset = @intCast(@offsetOf(Vertex, "pos")),
        },
        .{
            .location = 1,
            .binding = 0,
            .format = vulkan.VK_FORMAT_R32G32B32_SFLOAT,
            .offset = @intCast(@offsetOf(Vertex, "color")),
        },
    };
}
```

These functions are the Zig equivalents of the C++ static methods:

```cpp
Vertex::getBindingDescription();
Vertex::getAttributeDescriptions();
```

The important Zig mappings are:

| C++ | Zig |
|---|---|
| `sizeof(Vertex)` | `@sizeOf(Vertex)` |
| `offsetof(Vertex, pos)` | `@offsetOf(Vertex, "pos")` |
| `std::array<..., 2>` | `[2]vulkan.VkVertexInputAttributeDescription` |
| `array.data()` | `array[0..].ptr` |

The Vulkan fields retain their translated C names, such as `.inputRate`, `.location`, `.format`, and `.offset`.

### 4. Replace the empty vertex input state helper

The existing koba code currently creates an empty vertex input state:

```zig
fn initialize_vulkan_create_graphics_pipeline_vertex_input_state() vulkan.VkPipelineVertexInputStateCreateInfo {
    return .{
        .sType = vulkan.VK_STRUCTURE_TYPE_PIPELINE_VERTEX_INPUT_STATE_CREATE_INFO,
        .pNext = null,
        .flags = 0,
        .vertexBindingDescriptionCount = 0,
        .pVertexBindingDescriptions = null,
        .vertexAttributeDescriptionCount = 0,
        .pVertexAttributeDescriptions = null,
    };
}
```

Replace it with a helper that accepts the descriptions created by the pipeline setup code:

```zig
fn initialize_vulkan_create_graphics_pipeline_vertex_input_state(
    binding_description: *const vulkan.VkVertexInputBindingDescription,
    attribute_descriptions: []const vulkan.VkVertexInputAttributeDescription,
) vulkan.VkPipelineVertexInputStateCreateInfo {
    return .{
        .sType = vulkan.VK_STRUCTURE_TYPE_PIPELINE_VERTEX_INPUT_STATE_CREATE_INFO,
        .pNext = null,
        .flags = 0,
        .vertexBindingDescriptionCount = 1,
        .pVertexBindingDescriptions = binding_description,
        .vertexAttributeDescriptionCount = @intCast(attribute_descriptions.len),
        .pVertexAttributeDescriptions = attribute_descriptions.ptr,
    };
}
```

The pointers in this structure are not copied by Vulkan. They only need to remain valid until `vkCreateGraphicsPipelines` returns. Therefore, the binding and attribute arrays should be local variables in the function that creates the pipeline, not temporary values inside a helper call.

### 5. Update graphics pipeline creation

Inside the existing `initialize_vulkan_create_graphics_pipeline` method, locate the code that constructs the vertex input state. Replace the old zero-description setup with:

```zig
const binding_description = vertex_binding_description();
const attribute_descriptions = vertex_attribute_descriptions();

const vertex_input_state = initialize_vulkan_create_graphics_pipeline_vertex_input_state(
    &binding_description,
    attribute_descriptions[0..],
);
```

Then pass `&vertex_input_state` to the existing graphics pipeline create-info construction.

For example, if the current code calls a helper similar to:

```zig
const create_information = self.initialize_vulkan_create_graphics_pipeline_create_information(
    &rendering_information,
    &shader_stages,
    &vertex_input_state,
    &input_assembly_state,
    &viewport_state,
    &rasterization_state,
    &multisample_state,
    &color_blend_state,
    &dynamic_state,
);
```

the important part is that `&vertex_input_state` is now the populated structure shown above.

The relevant Vulkan pipeline create info must ultimately contain:

```zig
.vertexInputState = &vertex_input_state,
```

The exact surrounding arguments should remain the same as the existing koba implementation. This lesson changes only the vertex input portion.

### 6. Verify the resulting layout

The declarations above describe this byte layout:

```text
byte  0.. 7: pos.x, pos.y
byte  8..19: color.r, color.g, color.b
stride:        20 bytes
```

You can add a temporary debug log in the pipeline creation method:

```zig
std.log.debug(
    "Vertex layout: stride={d}, position_offset={d}, color_offset={d}",
    .{
        @sizeOf(Vertex),
        @offsetOf(Vertex, "pos"),
        @offsetOf(Vertex, "color"),
    },
);
```

This should report:

```text
stride=20, position_offset=0, color_offset=8
```

### 7. Keep the draw-path limitation in mind

The existing command recording code likely still records a draw based on the vertex ID:

```zig
vulkan.vkCmdDraw(command_buffer, 3, 1, 0, 0);
```

That call can remain for now, but the new shader expects vertex attributes. Until a vertex buffer is created and bound, the pipeline is only correctly *described*, not fully usable.

The next rendering step will add a vertex buffer and bind it before the draw:

```zig
vulkan.vkCmdBindVertexBuffers(
    command_buffer,
    0,
    1,
    &vertex_buffer,
    &vertex_buffer_offset,
);
```

At that point, the vertex count should come from the data rather than being hard-coded:

```zig
vulkan.vkCmdDraw(
    command_buffer,
    @intCast(vertices.len),
    1,
    0,
    0,
);
```

The buffer and its lifetime must be managed by `Application`, just like the existing swapchain, pipeline, and synchronization objects.

## Recap & What's Next

This lesson established the CPU-to-shader vertex layout contract:

- `Vertex` is represented as a C-compatible Zig `extern struct`.
- `@sizeOf` replaces C++ `sizeof`.
- `@offsetOf` replaces C++ `offsetof`.
- A binding description defines the record stride and input rate.
- Attribute descriptions map shader locations to Vulkan formats and byte offsets.
- The pipeline now describes position at location `0` and color at location `1`.
- The shader no longer generates triangle data from `SV_VertexID`.

The pipeline still needs an actual Vulkan vertex buffer. The next step is to allocate a buffer, upload the `vertices` array, bind it with `vkCmdBindVertexBuffers`, and draw using `vertices.len`.
